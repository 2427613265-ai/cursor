# 交底书创造性补强：跳出交底书，从三个方面提高创造性

## 适用场景

- 拿到一份**技术交底书**（发明人或客户提交的技术交底材料），准备据此撰写专利申请文件，或需要先评估、补强这份交底书时使用。
- 客户/发明人提交的交底书**往往并不完善**：常常只写了发明人自己最先想到的那一个具体方案，技术问题描述得笼统或表面，实现细节缺失，应用场景单一。如果撰写者只是把交底书内容照搬进专利，创造性会显得单薄，容易在审查阶段因"区别技术特征属于公知常识/显而易见"而被驳回或被迫大幅缩小保护范围。
- 本skill解决的是"**如何在忠于发明人真实技术贡献的前提下，跳出交底书本身、主动补充内容，把这件专利的创造性和授权可能性做上去**"这一问题。
- 核心态度：**不要局限于交底书给出的内容，不要被交底书局限思维**。交底书是撰写的起点而不是终点，撰写专利是一次"二次创造"。

## 与其他skill的衔接顺序

本skill位于整个撰写流程的**最前端**，紧接在拿到交底书之后、正式规划与落笔之前：

```
拿到交底书
  → 本skill（跳出交底书、补强创造性：重界定问题 / 细化方案 / 拓展场景）
  → 专利申请文件撰写策略（五维度整体布局）
  → 通用/说明书撰写（技术领域、背景技术、技术问题陈述、有益效果等固定段落）
  → 结构类/方法类/算法类（具体权利要求文字撰写）
  → 权利要求质检 / 说明书质检（定稿后复核）
```

本skill与同分支下的 `专利申请文件撰写策略` 是配合关系：本skill偏重"从交底书里挖出/补出更多有创造性的技术内容"，`专利申请文件撰写策略` 偏重"把已确定的技术内容按什么范围、防谁、怎么布局来落地"。建议先用本skill把技术内容补强到位，再用撰写策略skill做整体布局规划。

## AI 使用本skill的能力边界（最重要，先读这一节）

"补强创造性"绝不等于"编造技术事实"。二者的界限必须守住：

- **可以做的**：基于交底书已披露的技术特征和公知的技术原理，做**合理的技术推演**——例如推断某个结构的等效替代方式、某个步骤的可选参数区间、某个方案还能解决的其他技术问题、某个方案可迁移的其他应用场景。这类推演是撰写者"二次创造"的正常组成部分。
- **绝对不可以做的**：
  - 凭空杜撰交底书里没有、发明人也从未确认过的**结构/组件/算法步骤**，当作发明人已完成的技术方案写入正文；
  - 编造**具体的实验数据、参数数值、性能提升百分比、效果对比结果**；
  - 编造**现有技术检索结果或对比文件**（AI 不具备专利数据库检索能力）；
  - 把技术上**根本不可行或违背物理/数学常识**的"改进"当作创新点。
- **正确的工作方式**：本skill产出的所有补充内容，一律标注为"**建议补强方向**"，逐条交给用户/发明人确认——确认①技术上是否真实可行、②是否是发明人真实的技术构思或可接受的合理延伸、③是否补入权利要求/说明书。**未经发明人确认的推演内容，不得作为既定技术事实写入正式文本。** 发明人确认后，这些内容才从"AI 的推演建议"转化为"发明人认可的技术方案"，从而能被合法地写进说明书并获得说明书支持。
- 遇到关键信息缺失（如实现细节、参数、可行性）而发明人未提供时，应**主动向发明人提问**，而不是自己脑补填空。

> 一句话原则：**AI 负责"想到发明人没写全的方向"并提问，发明人负责"确认这些方向技术上成立"**。想象力用在提出方向上，事实以发明人确认为准。

## 补强创造性的三个着力点

### 着力点一：技术问题的界定

交底书里发明人对"要解决什么技术问题"的描述，常常有以下毛病：太笼统、太表面、太窄，或者只盯着一个显而易见的问题。而在创造性的判断中（审查采用"三步法"：确定最接近的现有技术 → 确定区别技术特征和发明**实际解决的技术问题** → 判断是否显而易见），**技术问题界定得是否恰当，直接影响区别技术特征是否被认定为"非显而易见"**。因此重新、准确地界定技术问题，是提高创造性最杠杆的一步。

补强做法：

- **深挖到更本质的技术问题**：发明人写的往往是"现象层"的问题（"效率低""不好用"），要追问其背后的技术根因，把技术问题界定到能体现方案巧妙之处、别人不容易想到的那个层次。技术问题界定得越贴合区别技术特征的独特作用，方案就越显得非显而易见。
- **挖掘"隐藏的"技术问题**：同一个方案可能同时解决多个层次的技术问题，其中一些是发明人自己都没意识到、但方案客观上确实解决了的。把这些一并识别出来，既能丰富说明书、支撑从权布局，也可能提供更有力的创造性论据。
- **让技术问题与区别技术特征相匹配**：技术问题应当是"由区别技术特征所实际解决的那个问题"，而不是背景技术里罗列的一堆缺陷的简单堆砌。撰写时要保证：独权的每个核心区别特征，都对应着一个被清楚界定的技术问题。
- **警惕把"目的/效果/非技术问题"误当技术问题**：技术问题必须是技术层面的问题，不能写成纯商业目的、纯主观感受或纯粹的效果陈述。

需要向用户/发明人确认的问题：
- [ ] 这个方案除了交底书写明的那个问题，实际上还顺带解决了哪些技术问题？
- [ ] 交底书描述的技术问题，其技术根因到底是什么？能否界定得更本质、更具体？
- [ ] 各个区别技术特征分别是针对哪个技术问题设计的？

自检：
- [ ] 重新界定后的技术问题，是不是比交底书原文更能凸显区别技术特征的非显而易见性？
- [ ] 每个被写进说明书/背景技术的技术问题，是否都能在权利要求里找到针对性解决它的技术特征？

（技术问题最终如何落到文字，参见同分支 `通用/说明书撰写` 下的 `说明书技术问题陈述撰写`、`说明书背景技术部分撰写`。）

### 着力点二：技术方案的细化

交底书通常只给出一个**粗略的方案骨架**，缺少实现细节、只有单一实施方式。细化技术方案能从多个角度提高创造性和专利质量：提供更多可布局为从属权利要求的区别特征，形成梯度防御；为独权的上位概括/功能性限定提供足够多的实施例支撑（否则得不到说明书支持）；满足"充分公开、能够实施"的要求；并可能挖出带来"意想不到的技术效果"的特征（意想不到的技术效果是主张创造性的有力论据）。

补强做法：

- **补充替代实施方式**：针对方案里的每个关键技术特征，追问"还能用什么别的结构/器件/步骤/算法实现同样的功能"，为上位概括和功能性限定积累实施例，也提前占位、封堵竞争对手的规避设计。
- **补充参数范围与优选值**：把交底书里的单点参数扩展为合理的取值范围，并标注优选值（但具体数值必须由发明人确认或来自其提供的实验，AI 不得编造数据）。
- **补齐实现细节，做到可实施**：把骨架方案展开到本领域技术人员照着就能实现的程度，补上被交底书省略的中间步骤、连接关系、控制逻辑。
- **挖掘机理与意想不到的技术效果**：分析每个特征"为什么有用"，识别出那些能产生超出预期效果的特征，重点论述其机理——这类特征往往是创造性的核心。
- **补充改进方案作为梯度与退路**：在核心方案之上提出进一步改进，既可布局为从权形成保护梯度，也可作为答复审查意见时修改权利要求的"退路"。

需要向用户/发明人确认的问题：
- [ ] 交底书里这个特征，实际上还有没有别的实现方式/替代结构？
- [ ] 这些参数的合理取值范围和优选值是多少？有没有实验数据支撑？
- [ ] 方案有没有可以进一步改进、性能更好的版本？
- [ ] 哪个特征带来了明显超出预期的效果？其技术机理是什么？

自检：
- [ ] 细化后的实施例数量，是否足以支撑独权里打算使用的上位概括/功能性限定表述？
- [ ] 补充的实现细节，是否达到了本领域技术人员据此可实施的程度？
- [ ] 补充的每一项实现细节/参数/效果，是否都已（或待）经发明人确认，没有 AI 凭空编造的成分？

（方案细化后如何落到权利要求文字与实施例，参见 `结构类/方法类/算法类` 分支及 `通用/说明书撰写` 下的 `说明书具体实施方式撰写`。）

### 着力点三：具体应用场景的拓展

交底书里发明人常常只想到**一个应用场景**（往往是他自己所在行业或最初的需求场景）。主动拓展应用场景可以：扩大保护范围与商业价值、体现技术方案的普适性；更重要的是，**方案迁移到新场景时往往需要做适应性调整，而这些调整本身可能构成新的创新点/新的技术特征/新的从权**。

补强做法：

- **横向迁移到其他领域/场景**：追问"这个方案的核心原理，除了交底书写的场景，还能用在哪里"，找出商业前景可能更好或更能体现普适性的场景（例如方案最初只针对某工业设备，其原理可能同样适用于医疗、消费电子等领域）。
- **重点找"能催生新技术特征或新技术问题"的场景**：拓展场景不是简单罗列越多越好——单纯堆砌场景对创造性帮助有限。真正有价值的是那些迁移过去会**倒逼方案做适应性改进**的场景，这些适应性改进可以布局为针对特定场景的从权，甚至独立权利要求。
- **评估是否需要为新场景单独布局**：若某个新场景商业前景显著且方案需较大调整，考虑是否针对该场景补充撰写方案或单独布局权利要求（与 `专利申请文件撰写策略` 维度一衔接）。

需要向用户/发明人确认的问题：
- [ ] 这项技术的核心原理，除了交底书描述的场景，还能应用到哪些其他领域？
- [ ] 迁移到这些新场景时，方案需要做哪些适应性调整？这些调整是否可行？
- [ ] 是否有某个新场景商业前景更好、值得为其单独布局权利要求？

自检：
- [ ] 拓展的场景里，是否至少有能催生新技术特征或新技术问题的场景，而不只是同一方案的简单换皮罗列？
- [ ] 为新场景补充的适应性改进，是否已交发明人确认可行、并考虑了是否布局为从权？

## 补强成果如何落地

三个着力点补强、并经发明人确认后，把成果分别落到专利申请文件的对应位置：

- 重新界定的技术问题 → 落到说明书"技术问题陈述""背景技术"，并保证与权利要求区别特征一一对应。
- 细化出的替代实施方式、参数范围、实现细节、意想不到的技术效果 → 落到说明书"具体实施方式""有益效果"，并把有价值的区别特征布局到从权（梯度防御 + 审查退路）。
- 拓展的应用场景及其适应性改进 → 落到说明书实施例，并视商业价值决定是否单独布局权利要求。

## 整体使用流程

1. 通读交底书，识别其"单一方案骨架 + 表面技术问题 + 单一场景"的不完善之处。
2. 依次过三个着力点（技术问题界定 / 技术方案细化 / 应用场景拓展），产出一份"**建议补强方向清单**"，逐条标注为待发明人确认。
3. 就清单中依赖发明人判断的项（可行性、真实构思、数据）**向用户/发明人提问确认**，不自行脑补填空，不编造技术事实或检索结果。
4. 确认后，把采纳的补强内容并入技术方案，再进入 `专利申请文件撰写策略` 的整体布局规划，随后正式撰写。
5. 全程守住能力边界：AI 负责"想到并提出发明人没写全的方向"，发明人负责"确认这些方向技术上成立"。

## 一句话总结

交底书是起点不是终点：从**技术问题的界定、技术方案的细化、具体应用场景的拓展**三个方面跳出交底书主动补强创造性，同时坚持"合理技术推演、不凭空杜撰、补充内容一律交发明人确认"，从而在忠于发明人真实贡献的前提下提高专利的创造性与授权可能性。
