# 专利申请文件撰写策略：五个维度

## 适用场景

- 在正式落笔撰写某一件专利申请具体权利要求/说明书文字**之前**，或撰写过程中需要跳出文字细节、重新审视整体方向时使用。
- 解决的是"这件专利应该往什么方向布局、覆盖哪些方案、防谁"这类**策略层**问题，不涉及具体某一句权利要求怎么措辞。
- 不区分结构类/方法类/算法类，五个维度对任意技术方案类型都适用；具体到某一类技术方案的文字撰写规范，参考同级的 `结构类`/`方法类`/`算法类` 分支，具体到说明书某一固定段落的写法，参考同级 `通用/说明书撰写` 分支。

## 与其他skill的衔接顺序

一次完整的专利申请撰写工作，建议按以下顺序调用相关skill，本skill排在最前面：

```
本skill（策略规划，明确方向）
  → 通用/说明书撰写（技术领域、背景技术等固定段落）
  → 结构类/方法类/算法类（具体权利要求文字撰写）
  → 专利质检/权利要求质检/patent-claims-quality-review（写完后整体复核）
```

即：先用本 skill 和用户/发明人对齐方向，再动笔写具体文字，写完后再用质检 skill 复核落地效果。不要跳过策略规划直接进入文字撰写，也不要把本该在规划阶段问清楚的问题拖到写完之后才发现。

## AI 使用本skill的能力边界（重要）

以下维度依赖的信息，AI 本身**没有能力独立获取**，只能依赖对话中用户已提供的内容做判断：
- 现有技术检索结果（AI 不具备专利数据库检索能力）
- 申请人的商业规划、目标市场、产品形态
- 竞争对手的产品/专利布局情报

**遇到信息不足时，应该向用户提出具体问题去问清楚，而不是凭空假设、脑补或编造**（比如编造并不存在的"检索结果"、编造申请人根本没提过的"商业前景"、编造凭空想象的"竞争对手改进方案"）。每个依赖外部信息的维度下面都列了"需要向用户确认的问题"，遇到用户没主动提供的信息，应主动提出这些问题，等用户回答后再据此调整撰写策略；如果用户明确表示"不需要考虑"或"没有更多信息"，则如实记录、按现有信息继续，不必反复追问。

## 维度一：商业前景与抵御方向

- 撰写之初就应该先搞清楚这件专利**为什么而写、想防住谁**——这决定了后续保护范围、布局策略的方向，应当最先明确。
- 和发明人/申请人沟通时，留意技术方案是否可以应用到比最初设想**更有商业前景**的领域（例如方案最初只考虑PC端，但发现移动端应用前景更可观）。一旦发现潜在的更优应用场景，要评估技术方案迁移到新场景是否需要调整，并据此及时调整撰写策略（例如是否需要针对新场景补充撰写方案或单独布局权利要求）。
- 明确希望通过这件专利**抵御的具体对象**是谁（生产商/销售商/使用者/平台方，或者是防止竞争对手抢先申请同类改进方案），不同的抵御对象会影响权利要求的设计侧重。

需要向用户确认的问题：
- [ ] 这项技术除了发明人最初描述的场景外，是否还有其他潜在应用领域？
- [ ] 这件专利主要希望通过维权/许可实现什么商业目的？希望抵御的对象是谁？
- [ ] 若技术方案可迁移到其他场景，是否需要针对该场景补充撰写内容？

## 维度二：适当的保护范围——检索在先、上位概括、多实施例支撑

- 撰写前应先充分检索国内外专利文献，摸清现有技术水平，据此确定权利要求的保护范围边界，让专利具备更高的稳定性——理想情况下，撰写者应该能大致"预测"审查阶段可能被检索出的对比文件。**AI 若没有检索能力或用户没有提供检索结果，应明确告知用户这一限制，建议用户补充检索结果或委托专利代理人检索，而不是假装已经检索过。**
- 撰写专利申请文件的过程是一次"二次创造"：需要和发明人一起讨论解决技术问题的**各种实施方式**，把具体方案的技术特征进行概括、抽象（上位概括或功能性限定），才能确定权利要求，而不是把发明人提供的单一具体方案照抄进独权。
- 上位概括/功能性限定需要**足够多的实施例**支撑，否则权利要求可能得不到说明书支持；功能性特征在侵权判定阶段会被解释为说明书披露的具体实施方式及其等同方式，而"是否等同"本身容易产生争议，所以撰写阶段就应该尽量挖掘更多的实施方式，为将来的解释留出空间。**如果用户只提供了一种实施例，应主动询问是否存在替代结构/参数范围/实现方式，不能为了凑数量而自行杜撰实施例。**

需要向用户确认的问题：
- [ ] 是否已经做过现有技术检索？能否提供检索到的对比文件或现有技术概况？
- [ ] 除了已经描述的方案外，是否还有其他可行的替代实施方式？
- [ ] 权利要求中打算使用功能性限定的地方，是否能提供不止一种具体实施方式？

自检：
- [ ] 说明书中的实施例数量是否足以支撑权利要求里的上位概括表述？

## 维度三：从有利于侵权判定的角度撰写

- 基于**全面覆盖原则**，独立权利要求只写解决技术问题所必需的最少技术特征集；任何非必要技术特征写进独权，只会不必要地缩小保护范围，应当放到从属权利要求中。
- 容易被忽视、但实际会缩小保护范围的写法，撰写时要仔细斟酌是否真的必要：
  - 发明名称本身带来的隐含限定
  - 使用环境特征
  - 以制备方法界定产品的技术特征（product-by-process）
  - 方法权利要求中的步骤顺序限定
- 权利要求中如果出现了容易产生歧义、而说明书里又没有解释的术语，会给日后侵权判定阶段对权利要求的解释造成障碍——应当在说明书中提前给出清楚的定义。

自检：
- [ ] 独权中的每一个技术特征，是否都是解决其声称技术问题所必需的？删掉是否仍不影响创造性？
- [ ] 是否存在发明名称、使用环境限定、product-by-process、方法步骤顺序等可能不必要缩小保护范围的写法？是否已仔细斟酌其必要性？
- [ ] 权利要求中有没有可能引起歧义的术语？说明书中是否已给出明确定义？

## 维度四：是否容易被竞争对手规避

- 与申请人一起主动拓展两类方案：
  - **改进技术方案**：可以作为专利审查阶段应对审查意见的"退路"（用于修改权利要求或补充为从权），同时也能提前占位、阻止竞争对手把这个改进方案抢先申请专利。
  - **变劣技术方案**：根据北京市高级人民法院《专利侵权判定指南》，变劣技术方案不构成等同侵权，因此撰写时也要一并考虑覆盖变劣方案，避免竞争对手用"故意变劣"的方式规避侵权。
- **AI 提出的改进方案/变劣方案，本质是基于已知技术特征的合理推演，不是凭空捏造的技术事实**，提出后应明确标注为"建议方向"，请用户/发明人确认技术上是否可行、是否需要补充到说明书或从权中，不能未经确认就直接当作既定方案写入正式文本。

需要向用户确认的问题：
- [ ] 针对当前方案，是否已经讨论过可能的改进方向？是否需要布局为从权或记录在说明书里作为审查阶段的"退路"？
- [ ] 是否讨论过可能的"变劣"替代方案（性能更差但仍能解决同一技术问题的替代做法）？是否需要额外的从权覆盖？

示例：

> 某案的核心技术方案是"用两个固定间距的传感器测量物体移动速度"。围绕这个方案可以提出：
> - **改进方案**（可作为从权/审查退路）：把"固定间距"改为"可调节间距"，覆盖了竞争对手换成可调间距设计来规避的可能性。
> - **变劣方案**（提请用户确认是否需要额外覆盖）：把"两个传感器"减少为"一个传感器+一次往返测量"，测量精度更差但仍能解决"测速"这个技术问题——这种变劣方案不构成等同侵权，如果不在从权里覆盖，竞争对手完全可以用这种"降级版"实现方式规避侵权。
>
> 这两类方案都应该提出来交给用户判断是否采纳，而不是自行断定后直接写入独权或从权正文。

## 维度五：布局策略

- 重视**从属权利要求**的价值：把独权之外、仍具备新颖性和创造性的技术特征布置到从权中，用于阻断竞争对手的规避设计，不要为了追求独权"简洁"而把有价值的区别特征全部舍弃。
- 当某些技术特征之间**互不兼容**（比如两种互斥的实施方案）时，可以考虑构建多条保护范围不同的独立权利要求分别覆盖，避免将来因为修改（例如答复审查意见时被迫删除或合并权利要求）而被迫放弃其中一部分保护范围。
- 方法权利要求和产品权利要求的保护范围可以**有意设计出层次差异**（例如方法权利要求的保护范围大于产品权利要求），以实现不同层次的保护，而不是让两者的保护范围完全重合。

自检：
- [ ] 从权是否已经覆盖了独权之外、具备新颖性和创造性的技术特征，用来阻断规避设计？
- [ ] 是否存在互不兼容的技术特征？是否需要拆分成多条独立权利要求分别保护，而不是勉强合并成一条？
- [ ] 方法权利要求与产品权利要求的保护范围，是否经过有意识的层次化设计？

## 使用建议

在正式撰写某一件专利申请的具体文字之前，建议按以上五个维度逐一过一遍：维度一、二、四涉及的问题应主动向用户/发明人确认清楚，不能自行假设或编造；维度三、五主要依据已确认的技术方案本身即可判断，可以自主执行。规划完成后再进入具体的权利要求和说明书撰写阶段（参考同分支下的 `结构类`/`方法类`/`算法类` 以及 `通用/说明书撰写` 目录下的 skill），写完后用 `专利质检/权利要求质检/patent-claims-quality-review` 做整体复核。
