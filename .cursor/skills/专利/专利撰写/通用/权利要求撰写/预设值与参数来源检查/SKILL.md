# 预设值与参数来源检查（机器人视角）

## 核心思想：把自己当作一个只会照做的机器人

权利要求是一份要交给"机器人"逐字执行的技术方案说明书。这个机器人**没有任何常识补全能力**，它只会严格按照权利要求写出来的文字，一步一步执行——权利要求没写的数据，它就拿不到；权利要求没交代来源的参数，它就无从获得。

因此，权利要求里每出现一个**预设值、阈值、具体参数、系数、比例、目标值、基准值、参考数据**等，撰写者都要停下来，站在机器人的立场连问三个问题：

1. **这个数据是权利要求里已经给定 / 已知的吗？**（比如是权利要求前文已经设定的、或本领域公认的固定常量）
2. **如果不是给定的，我（机器人）能直接获得它吗？**（比如通过一个已经写明的传感器、已经写明的计算步骤直接得到）
3. **如果两者都不成立**——即这个数据既不是已知的、机器人也没法凭现有步骤拿到——那么权利要求**必须先补上"获取 / 测量 / 采集 / 计算 / 设定 / 读取"这个参数的步骤**，然后才能在后续步骤里使用它。

**一句话规则：任何被"使用"的数据，在权利要求里必须先有明确的"来源"（给定、获取或算得），不能凭空出现、直接就用。**

## 三种数据来源，各自的合规写法

| 数据类型 | 是否需要补充获取步骤 | 说明 |
|----------|---------------------|------|
| **已给定/预先设定的值**（如"预设阈值T"） | 需要交代它是"预先设定的"，并尽量说明设定依据/范围 | 允许作为已知量直接使用，但要让机器人知道"这是一个预设量"，而不是需要它现算的量 |
| **本领域公认的固定常量**（如重力加速度、水的密度） | 不需要 | 机器人可视为已知，属于公知常识 |
| **依赖运行时才能确定的量**（如"实时采集的温度""当前转速""检测到的距离"） | **必须**先写明获取/测量/计算步骤 | 机器人执行前并不掌握这个数据，必须先有一步把它拿到 |

## 判断方法（撰写时随手自查 / 质检时逐条核对）

对权利要求中每一个数值、参数、变量，执行以下流程：

```
发现一个参数 X
  → 问1：X 是权利要求前文已经"预先设定"或"给定"的吗？
        是 → 检查是否已明确写出"预设/给定"字样，合规；
        否 → 进入问2
  → 问2：X 是本领域公认的固定常量（公知常识）吗？
        是 → 可直接使用，合规；
        否 → 进入问3
  → 问3：在使用 X 的这一步之前，权利要求里有没有一步明确"获取/测量/采集/计算/读取"了 X？
        有 → 合规；
        没有 → 缺陷：X 凭空出现、来源缺失，必须补上获取 X 的前置步骤，或说明它是预设值。
```

## 正确 / 错误示范

### 示范一：实时参数缺少获取步骤

```
错误：
一种温控方法，其特征在于，包括：
  当温度超过预设阈值时，启动制冷模块。
```

机器人视角的问题：**"温度"这个数据从哪来？** 权利要求从头到尾没有一步告诉机器人去"测量温度"，机器人手上根本没有温度值，无法执行"当温度超过预设阈值时"这个判断。（另外"预设阈值"虽然点明了是预设的，但如果全文没有任何设定/取值说明，也偏弱，见示范二。）

```
正确：
一种温控方法，其特征在于，包括：
  通过温度传感器实时采集被控空间的温度；
  判断所述温度是否超过预设阈值，若是，则启动制冷模块。
```

补上了"采集温度"这一步，机器人才拿得到"温度"，后面的判断才可执行。

### 示范二：预设值来源不清

```
偏弱：
  ……当偏差大于系数K时，触发校正。
```

"系数K"是什么？机器人不知道它的值，也不知道去哪拿。

```
正确（作为预设值）：
  预先设定校正系数K；
  ……当所述偏差大于所述系数K时，触发校正。
```
或（作为算得的值）：
```
  根据所述历史数据计算得到校正系数K；
  ……当所述偏差大于所述系数K时，触发校正。
```

### 示范三：算法类中间量凭空出现

```
错误：
  根据置信度对候选结果排序，输出置信度最高的结果。
```

机器人视角：**"置信度"是怎么来的？** 没有任何一步计算出各候选结果的置信度。

```
正确：
  对每个候选结果，计算其对应的置信度；
  根据所述置信度对所述候选结果排序，输出置信度最高的结果。
```

## 与其他规范的关系

- 本 skill 是 `权利要求清楚性与逻辑一致性规范` 中"孤岛特征""突然出现的术语"两条规则在**数据/参数**维度上的具体化和强化：孤岛特征关注的是"特征有没有后续呼应"，本 skill 关注的是"数据有没有前置来源"，方向相反、互补。
- 方法类/算法类权利要求中，参数获取步骤往往还涉及条件判断，配合 `条件判断语句规范` 一起检查。
- 若参数本身是非公知的自定义名词（如"调节比""置信度"），其**含义定义**是否就地写清由 `自定义名词就地定义` 负责；本 skill 负责的是它的**来源**是否交代。两者都要过一遍。
- 主要适用于**方法类、算法类**权利要求（步骤按时间顺序执行，"先获取后使用"的次序要求最强）；结构类权利要求若涉及"预设参数/设定值"，同样要交代该参数是预设的还是可调的。

## 检查清单

- [ ] 权利要求中每一个数值、阈值、参数、系数、比例、目标值，是否都能回答"它从哪来"？
- [ ] 依赖运行时才能确定的量（温度、转速、距离、置信度等），在被使用之前，是否有明确的获取/测量/计算步骤？
- [ ] 预设值/给定值是否已明确写出"预设""预先设定""给定"等字样，让机器人知道无需现算？
- [ ] 是否存在既非公知常量、又没有获取步骤、却被直接拿来判断或计算的"凭空参数"？
- [ ] 步骤顺序是否满足"先获取、后使用"——获取参数的步骤是否排在使用该参数的步骤之前？

## 应用范围

撰写权利要求时应边写边用"机器人视角"自查每一个参数的来源；对已撰写完成的权利要求书做质检时，本 skill 作为 `权利要求质检/权利要求书质检` 的专项检查项（尤其针对方法类/算法类案件）纳入使用。
