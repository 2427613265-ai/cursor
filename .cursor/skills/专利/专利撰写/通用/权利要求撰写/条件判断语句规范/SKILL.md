# 专利条件判断语句规范

## 核心规则

在撰写方法类或算法类权利要求时，涉及条件判断的步骤必须使用规范的判断语句格式。

### 基本格式

**单一条件判断（带else分支）：**
```
判断是否满足[条件描述]，若是，则[执行操作A]；若否，则[执行操作B]
```

**单一条件判断（仅true分支）：**
```
判断是否满足[条件描述]，若是，则[执行操作A]
```

### 禁止使用的表达方式

❌ **错误示范：**
- "如果满足条件A，那么执行操作B"
- "当条件A成立时，执行操作B"
- "在条件A的情况下，进行操作B"
- "检测是否满足条件A，是的话执行操作B"
- "如检测到条件A，则执行操作B"

✅ **正确表达：**
- "判断是否满足条件A，若是，则执行操作B"

---

## 详细规范

### 1. 基本条件判断

#### 仅有 true 分支

**格式：**
```
判断是否满足[条件]，若是，则[操作]
```

**示例：**
```
判断所述温度值是否大于预设阈值，若是，则触发报警信号
```

#### 包含 true 和 false 分支

**格式：**
```
判断是否满足[条件]，若是，则[操作A]；若否，则[操作B]
```

**示例：**
```
判断所述用户权限级别是否为管理员权限，若是，则允许访问；若否，则拒绝访问并记录日志
```

### 2. 多条件判断

#### 与关系（AND）

**格式：**
```
判断是否同时满足[条件1]和[条件2]，若是，则[操作]
```

或更明确地：
```
判断是否满足[条件1]且满足[条件2]，若是，则[操作]
```

**示例：**
```
判断是否同时满足所述温度值大于30℃和所述湿度值小于50%，若是，则启动降温装置
```

```
判断是否满足所述用户已登录且所述会话未超时，若是，则允许继续操作；若否，则跳转至登录页面
```

#### 或关系（OR）

**格式：**
```
判断是否满足[条件1]或[条件2]，若是，则[操作]
```

**示例：**
```
判断是否满足所述电量低于10%或所述设备温度高于安全阈值，若是，则进入保护模式
```

#### 复杂组合条件

对于复杂的多条件组合，保持"判断是否满足..."的句式，条件部分可以适当使用括号和连接词：

**示例：**
```
判断是否满足（所述用户权限为高级权限或所述用户属于管理员组）且所述操作类型为敏感操作，若是，则要求二次验证；若否，则直接执行
```

### 3. 嵌套条件判断

当需要表达嵌套的条件判断时，应拆分为多个独立步骤，每个步骤保持独立的判断语句格式。

❌ **错误示范（单步骤内嵌套）：**
```
S3：判断是否满足条件A，若是，则判断是否满足条件B，若条件B也满足则执行操作C，否则执行操作D
```

✅ **正确示范（拆分为多个步骤）：**
```
S3：判断是否满足条件A，若是，则进入步骤S4；若否，则进入步骤S6；
S4：判断是否满足条件B，若是，则执行操作C；若否，则执行操作D；
S5：...
S6：...
```

## 检查清单

在撰写或审核涉及条件判断的权利要求时，逐项检查：

- [ ] 所有条件判断是否都使用"判断是否满足..."开头
- [ ] true分支是否都使用"若是，则..."
- [ ] false分支（如有）是否都使用"若否，则..."
- [ ] 是否存在"如果...那么..."、"当...时..."等口语化表达
- [ ] 多条件判断的逻辑关系（与/或）是否表达清楚
- [ ] 嵌套判断是否已拆分为独立步骤
- [ ] 条件描述是否明确、可验证（避免"合适"、"适当"等模糊词）

## 正确示例汇总

### 示例1：简单判断（仅true分支）
```
S3：判断所述检测值是否超过所述预设上限，若是，则生成警告信息；
```

### 示例2：完整判断（true和false分支）
```
S4：判断所述用户输入是否为有效格式，若是，则进入下一步处理；若否，则返回错误提示信息；
```

### 示例3：与条件
```
S5：判断是否同时满足所述距离小于安全距离和所述速度大于预设速度，若是，则触发制动系统；
```

### 示例4：或条件
```
S6：判断是否满足所述电池电量低于阈值或所述设备进入低功耗模式，若是，则关闭非必要功能模块；
```

### 示例5：复杂条件组合
```
S7：判断是否满足（所述时间段为工作时间且所述用户已认证）或所述请求标记为紧急请求，若是，则立即处理请求；若否，则将请求加入待处理队列；
```

## 错误示例与修正

### 错误1：使用"如果"句式
❌ **错误：**
```
S3：如果检测值超过预设上限，那么生成警告信息；
```

✅ **修正：**
```
S3：判断所述检测值是否超过所述预设上限，若是，则生成警告信息；
```

### 错误2：使用"当...时"句式
❌ **错误：**
```
S4：当用户权限为管理员时，允许执行删除操作；
```

✅ **修正：**
```
S4：判断所述用户权限是否为管理员权限，若是，则允许执行删除操作；
```

### 错误3：口语化表达
❌ **错误：**
```
S5：检查温度是不是太高了，是的话就打开风扇；
```

✅ **修正：**
```
S5：判断所述温度值是否高于安全阈值，若是，则启动散热风扇；
```

### 错误4：混用表达方式
❌ **错误：**
```
S6：判断是否满足条件A，如果是，则执行操作B；
```

✅ **修正：**
```
S6：判断是否满足条件A，若是，则执行操作B；
```

### 错误5：条件和操作混在一起
❌ **错误：**
```
S7：检测到异常时发送通知；
```

✅ **修正：**
```
S7：判断是否检测到异常，若是，则发送通知；
```

---

## 注意事项

1. **"判断"是必需的开头词**：所有涉及条件判断的步骤都必须以"判断是否满足..."或"判断是否..."开头。

2. **"若是"和"若否"是固定搭配**：不要使用"如果是"、"要是"、"假如"、"倘若"等其他词汇。

3. **条件必须可验证**：条件描述应当是明确、客观、可验证的，避免使用"合适"、"适当"、"较好"等主观或模糊的表达。

4. **保持步骤独立**：每个步骤应该是独立、完整的操作单元，复杂的嵌套判断应拆分为多个步骤。

5. **适用范围**：本规范主要适用于方法类和算法类权利要求，结构类权利要求一般不涉及条件判断。

## 与其他规范的配合

本规范应与以下技能配合使用：
- `权利要求清楚性与逻辑一致性规范` - 权利要求的清楚性与逻辑检查
- `权利要求形式合规性规范` - 权利要求的形式合规检查
- `独立权利要求必要技术特征检查` - 必要技术特征检查

在撰写完成后，应先应用本规范检查判断语句格式，再应用上述技能进行全面质检。
