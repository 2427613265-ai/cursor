# 权利要求清楚性与逻辑一致性规范（所有案件类型通用）

## 适用场景

- 撰写权利要求的过程中，每写完一条就应该顺手对照本 skill 自查，而不是全部写完才回头检查。
- 对已撰写完成的权利要求书做整体质检时，本 skill 是 `权利要求质检/patent-claims-quality-review` 检查清单的重要组成部分。
- 本 skill 对应专利法关于权利要求"清楚、简要"的要求，检查的是**表达和逻辑层面**的问题；与 `patent-claims-formal-compliance`（引用编号、多项从权等格式规则）是两个不同维度，两者都应纳入质检。

## 规则一：孤岛特征——仅出现一次、前后都未呼应的特征

**什么是孤岛特征**：权利要求中某个技术特征（部件、参数、动作）只出现了一次，前面没有铺垫、后面也没有任何权利要求或本条内部的其他表述再利用它、依赖它、或对它做进一步限定——这个特征像一座"孤岛"，游离在权利要求的技术方案之外。

**不要轻易判定为孤岛的例外情况**：
- **公知术语**：本领域技术人员一看就懂、无需进一步说明其作用或连接关系的常规部件/概念（比如"螺栓""电机""处理器"），即使只出现一次也不算孤岛，因为它的作用是不言自明的。
- **暗含于后续操作或说明书中的特征**：如果该特征虽然只在权利要求里出现一次，但它的作用能从上下文的后续动作中自然推出（比如"设有转轴"之后紧跟"绕所述转轴转动"，"转轴"的作用已经通过后续动作体现），或者说明书里对它的功能有清楚说明，也不算孤岛。

**判断方法**：对权利要求中每一个技术特征提问——"这个特征后面还有没有被用到（被限定、被引用、被作为其他特征的连接对象）？如果没有，它是不是公知术语，或者它的作用能不能从紧邻的上下文动作中直接看出？"三个问题都是"否"，才判定为孤岛特征。

**发现孤岛特征后怎么处理**：不要直接删除了事——先判断这是撰写疏漏（本该有后续呼应但漏写了，比如漏了这个特征该起的连接/限定作用）还是这个特征本身确实不必要（可以删除而不影响方案完整性），向用户说明具体是哪种情况，不要擅自替用户做删除决定。

示例：
```
一种XX装置，其特征在于，包括壳体、电机和调节杆；所述电机安装于所述壳体内；所述调节杆一端伸出所述壳体外，用于手动调节。
```
"调节杆"提到了"用于手动调节"，但"调节"的对象是什么、调节的是哪个已定义的特征，权利要求全篇再没有出现——这就是一个孤岛特征，需要补充说明"调节杆"具体调节的是什么（比如连接到某个可调部件），或者删除这个语焉不详的部件描述。

## 规则二：突然出现且不是客观清楚物理量的特征

区分两类"突然出现"的特征：
- **客观清楚的物理量**：长度、角度、温度、压力、转速等本领域公认、有明确度量方式的物理量，即使权利要求没有专门定义，本领域技术人员也能准确理解其含义，不需要额外解释。
- **人为定义的词**：撰写者自己创造或赋予特殊含义的概念（比如"调节比""稳定系数""响应等级"这类不是通用物理量、而是撰写者根据本发明的具体结构人为定义出来的术语）。这类词如果在权利要求中突然出现、缺乏必要解释（既没有在权利要求前文给出定义，说明书里也没有明确的计算方法/定义方式），就构成了保护范围不清楚。

**判断方法**：遇到权利要求里的每一个"量"或"比例""系数""等级"类表述，先问"这是本领域公认的标准物理量，还是本发明特有的人为定义概念？"如果是后者，检查权利要求或说明书是否给出了明确的定义或计算方式；没有的话就是需要补充解释的缺陷。

示例：
```
不清楚："所述调节比大于1.5时，……"（"调节比"是什么与什么的比值？未定义）
清楚："所述调节杆的伸出长度与所述壳体高度的比值（调节比）大于1.5时，……"
```

## 规则三：技术概念模糊或含义不确定的用语

避免使用没有客观标准、因人而异的模糊表述，比如"大致""左右""明显""合理的""适当的""容易地""较好地"这类相对性、主观性词汇，除非说明书或权利要求本身给出了可以量化或明确判断的标准。

**判断方法**：逐句扫描，遇到程度副词、相对性形容词，追问"这个词的边界在哪里？本领域技术人员能否根据这个词准确判断某个具体实施方式是否落入或落出保护范围？"如果答案模糊，就需要替换成更精确的表述，或者补充可判断的量化标准。

示例：
```
不清楚："所述连接件的长度适当，以便于安装"（"适当"没有边界，无法判断具体多长算适当）
清楚："所述连接件的长度大于所述壳体高度的一半，以便于安装"
```

## 规则四：权利要求不能有歧义或反例

- **歧义**：同一句权利要求文字，能被解读出两种或以上互不相同的技术含义（比如某个修饰语到底修饰前面哪个词不清楚，或者"和/或"这类连接词的范围不清楚）。
- **反例**：能构造出一个具体场景，字面上完全满足权利要求的文字描述，但明显不能解决背景技术指出的技术问题，或者明显不是发明人想要保护的方案——这说明权利要求的措辞留了漏洞，把不该保护的情形也框了进去（或者反过来，把该保护的方案排除在外）。

**判断方法（反例测试）**：读完一条权利要求后，尝试构造一个"钻空子"的场景——按照字面意思该场景确实满足所有限定，但明显违背发明目的。如果能构造出来，说明这条权利要求的表述需要补充限定条件把这个反例排除掉。

示例：
```
权利要求："一种XX装置，其特征在于，包括可移动的支撑件"
反例：如果"可移动"没有限定移动方向/移动方式，那么支撑件哪怕只能"被人徒手搬走"也字面满足"可移动"，但这显然不是发明真正想保护的技术方案（真正想保护的可能是"沿导轨滑动"这种特定移动方式）——需要补充具体的移动方式限定，排除掉这种不合理的字面解读。
```

## 规则五：描述需要清楚、简洁

- 避免不必要的重复表述、冗长的修饰语堆砌；能用一个技术特征说清楚的，不要拆成好几句绕来绕去。
- 避免同一个意思用不同的句式在权利要求书的不同位置反复强调（比如多条从权都在重复交代同一个已经清楚的限定关系）。
- 每个技术特征的表达应当只有一种自然的理解方式，读完权利要求应该能在脑海中直接构建出对应的技术方案，而不需要反复琢磨句子结构。

**判断方法**：通读整条权利要求，如果需要读两遍以上才能确定某个短语具体修饰的是谁、限定的是什么，就说明表达不够清楚简洁，需要重新组织句子结构或拆分成多个短句。

## 规则六：不能违背物理/数学常识，或前后逻辑矛盾

- **违背常识**：权利要求描述的技术方案在物理上不可能实现，或者违反基本数学关系（比如声称某个封闭结构的"体积"随某个线性参数"周长"的一次方增大，量纲关系明显有误）。
- **前后逻辑矛盾**：同一条权利要求内部、或者从权与它所依附的权利要求之间，存在互相矛盾的限定（比如独权说某部件"固定连接"，从权又说该部件"可拆卸"却没有说明是对固定连接方式的进一步限定还是矛盾表述；或者两处使用同一量纲的参数，取值范围却互相排斥）。

**判断方法**：
1. 对涉及物理量、数量关系的表述，检查量纲和数量级是否合理，是否符合基础物理/数学常识。
2. 对每一条从权，检查它新增的限定和它所依附的权利要求（及更上级的权利要求）的限定之间，是否存在互相排斥、无法同时成立的情况。

## 与已有检查项的关系

以下两点如果没有被本skill其他规则覆盖，仍然需要检查（属于清楚性/逻辑一致性范畴，和上面六条规则并列）：
- **术语前后一致**：同一个技术特征自始至终只用同一个名称指代，不能一会儿叫"夹角"一会儿叫"倾斜夹角"、一会儿叫"基准角度"。
- **引用编号与术语来源对应**：每个从权引用的权利要求编号，是否真的是该权利要求用到的每个名词第一次出现的地方（这一点也和 `patent-claims-formal-compliance` 的引用方向规则相关，但那是纯编号大小的形式核对，这里是"名词出处是否对得上"的语义核对，两者都要做）。

## 检查清单

- [ ] 权利要求中每一个技术特征，是否都能在后文找到呼应，或者属于公知术语/暗含于后续操作？有没有游离的孤岛特征？
- [ ] 是否存在人为定义、非客观物理量的术语，且没有给出明确定义或计算方式？
- [ ] 是否存在"大致""适当""合理"这类没有客观边界的模糊表述？
- [ ] 能否构造出字面满足权利要求、但明显不是发明真正想保护的反例场景？权利要求是否存在可能被解读出多种含义的歧义表达？
- [ ] 整条权利要求读起来是否清楚、简洁，是否存在冗余重复的表述？
- [ ] 涉及物理量/数量关系的表述，是否符合基本物理/数学常识？是否存在前后（含从权与其依附对象之间）互相矛盾的限定？
- [ ] 同一技术特征全篇是否用了统一的名称？
- [ ] 从权引用的权利要求编号，是否真的是相关名词第一次出现的地方？

## 应用范围

本规则适用于任意技术方案类型（结构类/方法类/算法类）权利要求的撰写与质检。撰写时应边写边对照检查，而不是全部写完才回头统一核查；质检时作为 `权利要求质检/patent-claims-quality-review` 检查清单的组成部分之一使用。
