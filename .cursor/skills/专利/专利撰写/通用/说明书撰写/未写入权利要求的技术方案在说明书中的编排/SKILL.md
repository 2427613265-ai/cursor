# 未写入权利要求的技术方案在说明书中的编排（所有案件类型通用）

## 适用场景

按 `说明书具体实施方式撰写` 逐条写完各条权利要求（独权+从权）对应的具体实施例之后，手头往往还有一批**没有写进权利要求、但确实存在于技术方案中**的内容，主要来自三个来源：

1. **交底书里提供的**：发明人交底时给出、但撰写权利要求时没有纳入（可能因为不属于必要技术特征、或保护范围取舍的考虑）；
2. **AI补充的**：撰写过程中由AI扩展、联想出的可选实现方式、优选参数、替代结构；
3. **人为补充的**：撰写人根据经验补充的改进、变形、应用场景。

这些"未入权技术方案"**同样要写进说明书"具体实施方式"部分**，本skill只解决一个问题：**它们应该放在哪里**。

## 核心规则

### 为什么未入权的方案也要写进说明书

- **充分公开**：说明书的作用是把发明讲透，未入权但真实存在的技术内容写进去，能让公开更充分。
- **为后续修改/分案留支持基础**：审查过程中若需要把某个特征补进权利要求（缩小保护范围以获得授权），或后续提分案申请，被补入的内容**必须原本就记载在说明书里**，否则会因"超出原始申请文件记载的范围"而不被允许。写进说明书，等于给将来的权利要求修改预留了合法的支持基础。
- 因此，不能因为"这条没写进权利要求"就把它从说明书里丢掉——**丢掉的是将来可用的弹药**。

### 编排位置：强相关就近放，独立的放最后

写完权利要求的具体实施例后，对每一条未入权技术方案，先判断它与哪条权利要求的关系，再决定位置：

1. **与某条权利要求技术方案强相关的补充方案 → 紧跟在该条权利要求的具体实施例之后。**
   - 判断标准：这条补充方案是对某条权利要求里某个部件/步骤/参数的进一步细化、优化、替代实现或优选取值，**依附于该权利要求的技术特征**而存在，脱离那条权利要求就没有意义。
   - 就近编排的好处：读者读完该权利要求实施例后，紧接着就能看到围绕同一技术点的延伸方案，技术脉络连贯，也清楚体现这个改进与权利要求方案之间的配合关系。

2. **独立的、关联不大的技术方案或改进 → 集中放到整个"具体实施方式"的最后。**
   - 判断标准：这条方案针对的是权利要求没有涉及的**另一个部件、另一项功能或另一个应用场景**，即使把它删掉，也不影响前面任何一条权利要求实施例的完整性。
   - 放最后的好处：不打断前面权利要求实施例的主线叙述，把边缘性、独立性的内容集中收纳，结构清晰。

### 与权利要求实施例主线严格区分

无论放在哪里，未入权技术方案都要用引导语明确标注为**可选/补充**方案，例如"**在一个可选的实施方式中**""**进一步地**""**作为一种改进**""**在另一实施例中**"，与权利要求实施例的"复述+具体而言展开"主线区分开。

这一点与 `说明书具体实施方式撰写` 里"不能夹带新特征"的要求呼应：这些补充方案**绝不能混进权利要求实施例的复述段或"具体而言"展开段里**，冒充权利要求本身就有的技术特征——那样会造成实施例与权利要求的技术特征错位。补充方案必须另起，明示是可选内容。

## 判断"强相关"还是"独立"的方法

对每条未入权方案自问：

- "它是在细化/优化前面**某一条**权利要求已经出现过的部件或步骤吗？" → 是，则强相关，放那条实施例后面；
- "它讲的是前面权利要求都没提过的、另一个独立的部件/功能/场景吗？删掉它前面的实施例照样完整吗？" → 是，则独立，放整段最后。

拿捏不准、既能挂靠某条权利要求又有一定独立性的，优先按"强相关就近放"处理，让技术脉络更连贯。

## 示范（承接无人机马达散热外壳案例）

假设权利要求实施例已按权1、权2……逐条写完，交底书/补充里还有两类内容：

- **强相关**：交底书提到"导流齿的齿顶可采用圆角以进一步降低风阻"——这是对权利要求里"导流齿"这一部件的进一步优化，依附于导流齿存在。
  → 紧跟在涉及导流齿的那条权利要求实施例之后，写成："在一个可选的实施方式中，所述导流齿9的齿顶设置为圆角，以进一步降低马达高速旋转时的空气阻力与涡流噪音。"

- **独立**：交底书提到"外壳表面可涂覆纳米疏水涂层以提升防水溅性能""中心安装盘可设置通用安装孔以适配不同型号电机"——这些讲的是表面处理、安装适配，与权利要求主线的"导流散热结构"关系不大，删掉也不影响前面实施例完整。
  → 集中放到整个"具体实施方式"的最后，写成："进一步地，在另一实施例中，所述散热外壳的表面还可涂覆纳米疏水涂层……；所述中心安装盘上还可开设通用安装孔……"

## 检查清单

- [ ] 交底书提供的、AI补充的、人为补充的但未写进权利要求的技术方案，是否都已写进说明书"具体实施方式"，没有被丢掉（保证充分公开、留足后续修改/分案的支持基础）？
- [ ] 强相关的补充方案，是否紧跟在其所依附的那条权利要求的具体实施例之后？
- [ ] 独立的、关联不大的技术方案或改进，是否集中放到整个"具体实施方式"的最后？
- [ ] 每条补充方案是否用"在一个可选的实施方式中/进一步地/作为一种改进/在另一实施例中"等引导语明示为可选内容？
- [ ] 有没有把补充方案混进权利要求实施例的复述段或"具体而言"展开段里，冒充权利要求本身的技术特征？
- [ ] 补充方案本身的结构、参数、作用是否也描述清楚（参照 `说明书质检/说明书具体实施方式清楚性与逻辑一致性规范`）？

## 应用范围

适用于任意技术方案类型（结构类/方法类/算法类）说明书"具体实施方式"部分的撰写与质检。与 `说明书具体实施方式撰写` 配合使用：先按那条 skill 逐条写完各权利要求的"复述+展开"实施例，再按本 skill 把未入权技术方案就近或集中编排进去；补充方案的清楚性，则按 `说明书质检/说明书具体实施方式清楚性与逻辑一致性规范` 核对。
