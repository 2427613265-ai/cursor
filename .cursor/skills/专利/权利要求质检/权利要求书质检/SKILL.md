# 权利要求书质检规范

本 skill 用于**整体复核**已经撰写完成的权利要求书。

## 与单条权利要求审查的选用指南

| 场景 | 使用哪个 skill |
|------|---------------|
| 用户指定检查**某一条**权利要求（如"权2""权利要求3"） | `单条权利要求审查` |
| 用户要求**质检/审核整份**权利要求书 | 本 skill |
| 单条权利要求修改后，需确认是否影响全文编号或引用布局 | 先用 `单条权利要求审查`；若涉及编号重排或引用关系变化，再按本 skill 做整份复核 |

## 标准质检流程与 skill 读取分级

按以下顺序执行。**必读** skill 每次整份质检都必须加载；**按需** skill 仅在案件类型或已发现问题涉及相应领域时再加载，避免重复读取无关规范。

```
必读（每次整份质检）：
  1. 权利要求形式合规性规范           → 形式合规硬伤
  2. 权利要求引用结构检查   → 平行/串联引用结构
  3. 本 skill 维度一至四                        → 实质与布局

按需深入：
  - 结构类案件或有装置/部件类从权
      → 结构类权利要求撰写、功能性限定使用边界规范
  - 方法类/算法类案件或含条件判断步骤
      → 条件判断语句规范
  - 权利要求涉及预设值、阈值、参数、系数、目标值等数据时
      → 预设值与参数来源检查
  - 权利要求出现非公知的、自定义的名词时
      → 自定义名词就地定义
  - 维度三发现指代、主谓或方位词问题时
      → 权利要求清楚性与逻辑一致性规范（总则）；必要时再读
        权利要求主谓宾结构规范、权利要求指代与参照清晰性规范
  - 独权范围需逐项核对某特征是否必要时
      → 独立权利要求必要技术特征检查
```

**执行顺序**：先用 `权利要求形式合规性规范` 排除形式硬伤 → 再用 `权利要求引用结构检查` 检查独立改进点是否应平行引用、依赖改进点是否应串联引用 → 最后按下面四个维度做实质审查。**优先输出可机械核对的硬伤**（形式违规、依附编号错误、引用结构不当），再讨论范围/创造性/梯度等需专业判断的问题。

## 维度一：独立权利要求——保护范围与创造性的平衡

1. **必要技术特征是否齐全**：独权是否包含了解决其声称的技术问题所必需的全部技术特征？少了任何一个，方案就不完整、可能得不到说明书支持。
2. **有没有可以删掉但不影响创造性的多余限定**：独权里每一个技术特征，如果删掉后方案仍然新颖、非显而易见、仍能解决背景技术指出的问题，这个特征就应该降级到从权，不要留在独权里不必要地缩小保护范围。可结合 `独立权利要求必要技术特征检查` 逐项核对。
3. **核心区别特征是否真的能支撑创造性**：与背景技术提到的最接近现有技术相比，独权的区别技术特征是不是本领域技术人员容易想到的常规变型？这个区别特征带来的效果是不是真正非显而易见？
4. **保护范围有没有被不必要的具体数值/具体实现方式绑死**：独权里如果出现了具体角度数值、具体一种机构（比如限位槽+限位销这种具体结构），要考虑这些是否该降级到从权，让独权保持更概括的表达。
5. **独权里保留的功能性限定/效果陈述，是否构成了不必要的隐含限定**：如果独权里写了"从而实现XX效果"，理论上会被解释为一个隐含的限定条件（即必须真正产生这个效果才落入保护范围），如果这是撰写者主动的取舍要如实告知，不是撰写者本意就要建议删除。

## 维度二：从属权利要求——依附关系与梯度防御布局

1. **依附编号是否正确——逐条机械核对**：对每一条从权，列出它这一条新用到的所有名词，反查这些名词分别是在哪一条权利要求里第一次出现的，取其中编号最大的那条，从权的依附对象必须等于或晚于（即依附链条上包含）这条。**只要从权用到了某个词，而依附链条里找不到这个词的出处，就是编号错误，必须改**。这是最容易在插入/删除/重排权利要求后出错、也是最容易被审查员一眼挑出来的问题，每次调整编号后都要重新全篇核对一遍，不能只抽查改过的那几条。
2. **引用结构是否合理——平行/串联选择**：按 `权利要求引用结构检查` 检查：相互独立的改进点是否平行引用同一基础权利要求；存在技术依赖的改进点是否正确串联引用；组合改进是否使用"或/任一项"式多项引用。不必要的串联引用会让读者误以为存在技术依赖，也会限制从权的独立主张空间。
3. **梯度是否合理——由概括到具体，由结构到数值**：好的布局应该是"先用一条从权概括性地限定技术手段 → 再用下一条限定其具体结构 → 再用更下一条限定具体数值/档位/锁定方式"。检查是否存在"一步到位塞入所有细节"（层次不够）或者"该合并的却拆得太细"（浪费权利要求数量）。如果几个并列分支的精细程度不对称（比如一个分支拆了三层、另一个分支只有两层），要指出来，由用户决定是否要为了对称性再调整。
4. **从权分支之间有没有实质重复**：检查是否存在两条从权本质上用的是同一种技术手段（哪怕依附对象、措辞不同），如果一条已经隐含覆盖了另一条的保护范围，后者对创造性没有增量贡献，应删除或改成确实不同的技术方案。
5. **并列的互斥实施例分支有没有交叉污染**：比如两种互斥的替代实施例（分支A、分支B）应该各自独立依附同一个上位权利要求，检查有没有从权错误地跨分支引用了另一分支才有的术语。

## 维度三：逻辑完善性与清楚性

以 `权利要求清楚性与逻辑一致性规范` 为总则检查清单；发现指代、主谓或方位词专项问题时，再按需加载 `权利要求主谓宾结构规范` 和 `权利要求指代与参照清晰性规范` 深挖。检查项包括：

- 孤岛特征（仅出现一次、前后都未呼应的特征，公知术语/暗含于后续操作的特征除外）
- 突然出现且不是客观清楚物理量的人为定义术语，是否给出了明确定义；**非公知的自定义名词是否在权利要求内部用"其中"句就地定义，而不是只写在说明书里**（详见 `自定义名词就地定义`）
- 技术概念模糊或含义不确定的用语（"大致""适当"等）
- 权利要求是否存在歧义或可以构造出反例
- 描述是否清楚、简洁
- 是否违背物理/数学常识，或前后（含从权与其依附对象之间）逻辑矛盾
- **参数来源（机器人视角）**：每个预设值/阈值/参数/系数在被使用前是否都有明确来源（给定、公知常量或已写明获取步骤），有无凭空出现、直接就用的"无源参数"——详见 `预设值与参数来源检查`
- 术语前后一致：同一个技术特征自始至终只用同一个名称指代
- 引用编号与术语来源要对应（与维度二第1点是同一件事，从"术语"和"编号"两个角度双重核对）
- 主谓宾结构：每个动作是否有明确执行者，避免因省略主语导致指代不明
- 指代词清晰性：优先使用"所述+完整名称"；存在多个同类对象时必须消歧
- 方位词参照物：周向/轴向/径向、平行/垂直等是否明确了参照物

## 维度四：表达清楚性（按案件类型按需检查）

- 结构类案件：`结构类权利要求撰写`——是否先总述部件、后展开结构
- 结构类案件：`功能性限定使用边界规范`——功能性限定/技术效果表述是否恰当
- 方法类/算法类案件：`条件判断语句规范`——条件判断步骤是否使用规范句式
- 含预设值/参数的案件：`预设值与参数来源检查`——每个参数在使用前是否都有明确来源，是否满足"先获取、后使用"
- 含自定义名词的案件：`自定义名词就地定义`——非公知自定义名词是否用"其中"句在权利要求内部就地定义

## 输出格式

质检结果按以下顺序输出：

1. 形式合规性硬伤（`权利要求形式合规性规范`）
2. 引用结构问题（`权利要求引用结构检查`：不必要的串联/错误的平行引用等）
3. 依附编号错误（逐条列出"第几条权利要求 → 应改为依附第几条 → 原因"）
4. 实质重复问题
5. 独权范围与创造性观察
6. 从权梯度布局观察
7. 表达清楚性核对（如有遗漏）
