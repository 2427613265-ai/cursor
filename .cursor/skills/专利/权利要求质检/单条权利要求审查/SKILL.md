# 单条权利要求审查

当用户提到检查、审阅或修改**某一条**权利要求（如"权1""权2""权利要求3"等）时，按以下六个维度逐一检查，逐条给出具体问题和修改建议，而不是笼统评价。若某维度未发现问题，也要明确指出"该维度未发现问题"。

**与整份质检的分工**：本 skill 聚焦**单条权利要求的实质审查**（含说明书支持、现有技术区分、策略观察）。涉及整份权利要求书的编号核对、引用结构布局、形式合规等系统性检查时，应改用 `权利要求书质检`，并按其"必读/按需"分级加载子 skill，避免与本 skill 重复展开相同检查项。

**重要前提**：权利要求经常先于说明书修改（说明书会在权利要求定稿后统一同步）。因此"说明书尚未记载该特征"本身**不算撰写缺陷**，不要作为问题扣分。只有当新增/修改的技术方案存在下列情况时才需要指出：技术方案本身不合理、自相矛盾、明显无法实现，或该特征依赖的原理在说明书中完全找不到任何相关铺垫（而不仅仅是措辞未同步）。其余情况一律归入"待同步说明书事项"清单单独列出，提醒用户后续同步，而非当作问题提出。

## 1. 逻辑完善性与保护范围

- 引用关系是否正确：从权引用的权利要求号是否存在、主题类型是否一致（如方法权利要求不应引用装置权利要求）。
- 是否遗漏解决技术问题所必需的技术特征，独权是否因此保护范围不稳固。可结合 `独立权利要求必要技术特征检查` 核对。
- 独权概括是否过宽（可能缺乏新颖性/创造性支持）或过窄（保护范围不必要地受限）。
- 从权是否层层递进、逐步限缩，是否存在从权与独权特征矛盾、重复或从权未实际缩小保护范围的情况。
- 格式类引用问题（引用方向、多项从权择一引用等）以 `权利要求形式合规性规范` 为准，本维度只关注该条权利要求层面的逻辑后果。

## 2. 表达清楚性

以 `权利要求清楚性与逻辑一致性规范` 为总则；指代、主谓、方位词问题按需参照 `权利要求指代与参照清晰性规范` 和 `权利要求主谓宾结构规范`。重点包括：

- 用词是否精确、无歧义，是否存在模糊限定词（如"较大""适当""大约"）
- 同一技术特征命名和表述是否前后一致
- "所述"使用是否规范
- 句式是否冗长、是否存在多重修饰导致的指代不清
- 功能性限定是否有说明书具体实施方式的支持

## 3. 撰写规范与格式

以 `权利要求形式合规性规范` 为准，本维度补充单条层面的观察：

- 独权是否为"一种……"等规范句式，是否包含前序部分和特征部分（以"其特征在于"分隔）
- 从权引用格式是否规范
- 主题名称是否与说明书、摘要对应
- 开放式（"包括"）与封闭式（"由……组成"）过渡词是否恰当
- 结构类从权引入多个新部件时，是否遵循 `结构类权利要求撰写`（先总述部件、后展开结构）
- 方法类/算法类条件判断步骤是否遵循 `条件判断语句规范`

## 4. 说明书支持与合法性

- 权利要求中每个技术特征、术语是否能在说明书中找到对应的定义和支持。**若仅是说明书暂未同步更新，记入"待同步说明书事项"清单，不作为缺陷**。
- 是否存在修改超范围的内容（新增特征在说明书原文中找不到任何依据，且并非本次待同步的正常修改）。
- 数值范围的两端是否有实施例支持（若尚未同步，记入待同步清单）。
- 是否掺杂非技术特征，是否需要转化为技术手段的表述以避免客体适格性问题。

## 5. 与现有技术的区分

- 权利要求是否明显落入已知的现有技术/背景技术，是否需要补充区别技术特征。
- 独权是否清楚体现相对于现有技术的技术贡献，便于后续创造性答辩。

## 6. 权利要求布局与维权策略

本维度侧重**单条权利要求在整体布局中的角色**；整份权利要求书的引用结构（平行/串联）和梯度防御，由 `权利要求书质检` 配合 `权利要求引用结构检查` 系统审查。

- 该条从权在保护梯度中的位置是否合理（由宽到窄）
- 若为方法权利要求，是否存在侵权取证困难，是否建议增加对应的产品/装置权利要求
- 该条是否覆盖独特的实施场景/规避设计角度，或与其他从权实质重复

## 输出格式

按维度分点列出问题，每条说明：具体位置/特征 → 存在的问题 → 修改建议。全部检查完后，如有必要可给出优先级最高的 1-3 条改进建议汇总。

若存在因权利要求先于说明书修改而产生的差异，单独列出"待同步说明书事项"清单（不计入问题列表），逐条说明需要在说明书中补充或调整的内容。

若审查中发现编号或引用关系可能影响全文，在末尾提醒用户按 `权利要求书质检` 做整份复核。
